# 导航与剧情管理

剧情管理是将你的故事/游戏划分为较小场景和/或子剧情的过程，
这高度取决于你的叙事风格以及运行时 IO 等技术要求。

导航是另一个你可能关心的概念。
它是你引导玩家（观众）穿越不同剧情和子剧情的方式，
让他们能够交互式地体验你的游戏/故事。

在本页中，我们将了解 Arrow 提供的基本工具来帮助规划你的故事。

## 场景和跳转

场景和图形节点系统本身是最基本的导航和剧情管理功能。

每个场景可以包含任意数量的（子）剧情，由多个链接的图形节点组成。

如果简单的图形链接不够用，或者你想在保持网格整洁的同时连接两个相距较远的节点，[Jump](../nodes/jump) 节点就会派上用场。

Jump 节点可以将玩家导航到任何（目标）节点，无论是在同一场景中还是在其他场景中。
它们没有任何出站连接；相反，它们接受另一个节点的名称作为其目标参数，
当播放时（在控制台或运行时中），通过播放该目标节点从另一个点继续剧情。

当没有目标节点时，播放将在该点结束（即 EOL：行结束）。

> 你可以毫无问题地重命名目标节点，Jump 节点使用内部 `UID` 来跟踪它们的目标
> 并自动适应新名称。

通常，我们建议保持场景独立且尽可能小。
这是一个良好的实践，使大型项目的开发更容易。

## 分支与合流

分支和合流是使交互式体验正常工作的主要组成部分。

通过分支剧情线，你为观众提供了不同的路径来探索，
其中一个决定可以改变故事或结局。

Arrow 的主要分支工具是 [Interaction](../nodes/interaction)、[Dialog](../nodes/dialog) 和 [Randomizer](../nodes/randomizer) 节点。

Interaction 和 Dialog 节点接受一个传入连接（或被跳转到）
并让用户从一组选项中选择。
每个选项都可以将用户导航到另一个节点，
即另一条可能的剧情线。

Randomizer 正是按照它们应该做的那样做。它们获得一个传入连接
并将玩家发送到随机选择的出站连接。

Arrow 还通过自定义逻辑提供更多方式来为你的叙事引入交互性；
即无代码编程，或将变量操作节点与条件节点结合使用。

> 要了解更多信息，请查看我们关于[变量](./variables-logic)和逻辑的文档页面。

合流或合并是分支的反面。
它是当你引导玩家最终到达一个确定性的焦点节点时。
合流的主要技术目的是通过减少可能性来限制或管理子剧情的复杂性。
换句话说，当你希望一些由不同决定开始的独立增长的故事事件序列
合并到一个特殊事件中，这样你就可以用可管理数量的可能未来继续故事的其余部分。

[Hub](../nodes/hub) 和 [Jump](../nodes/jump) 节点都可用于合流你的剧情。
Hub 将同一场景中的传入图形链接合并为一个输出，
而 Jump 在你想要合并来自多个不同场景的分支时非常有用。
